# Gateway Integration

**Module:** `medscribe.src.gateway.app`

MedScribe exposes two EHR integration endpoints via HealthChain's gateway infrastructure: a SOAP/CDA endpoint compatible with Epic's NoteReader and a REST endpoint following the CDS Hooks specification.

## Application Factory

```python
from medscribe.src.gateway.app import create_app

app = create_app(
    fhir_gateway=None,          # optional FHIRGateway for live FHIR searches
    fhir_source="epic",         # named FHIR source
)
```

`create_app()` builds the full application:

1. Creates a `MedicalCodingPipeline` (simple, NER-only)
2. Creates a `MedGemmaClientProtocol` client (mode from env)
3. Creates a `CdaAdapter` for CDA document parsing
4. Instantiates `MedScribeAgent` with all components
5. Creates a `HealthChainAPI` (FastAPI-based) with two service routers

The module also exposes a ready-to-use instance at module level:

```python
# medscribe/src/gateway/app.py
app = create_app()
```

This supports direct uvicorn execution:

```bash
MEDGEMMA_MODE=demo uv run uvicorn medscribe.src.gateway.app:app --port 8000
```

## Registered Routes

| Method | Path | Protocol | Description |
|--------|------|----------|-------------|
| GET/POST | `/notereader/` | SOAP | Epic NoteReader WSDL + document processing |
| GET | `/cds/cds-discovery` | REST | CDS Hooks service discovery |
| POST | `/cds/cds-services/medscribe-coding-review` | REST | CDS Hooks coding review |
| GET | `/health` | REST | Health check |
| GET | `/docs` | REST | OpenAPI interactive docs |
| GET | `/metadata` | REST | Server metadata |

## NoteReader SOAP Service

The NoteReader endpoint is compatible with Epic's NoteReader API, which sends clinical documents as CDA (Clinical Document Architecture) XML via SOAP.

### Request Flow

```
Epic NoteReader → SOAP POST /notereader/ → CDA XML
    → CdaAdapter.parse(request) → Document with clinical note text
    → MedScribeAgent.run(note_text) → result dict
    → CdaAdapter.format(doc) → CDA XML response
```

### Method: `ProcessDocument`

```python
@notereader.method("ProcessDocument")
def process_document(request: CdaRequest) -> CdaResponse:
    ...
```

- Receives a `CdaRequest` containing a CDA document
- Parses via `CdaAdapter` to extract the clinical note text
- Runs the full MedScribe agent pipeline
- Returns a `CdaResponse` with the processed document

### WSDL

HealthChain auto-detects the Epic service WSDL template at startup. The WSDL is served at `GET /notereader/?wsdl`.

## CDS Hooks Service

The CDS Hooks endpoint follows the [HL7 CDS Hooks specification](https://cds-hooks.org/), enabling MedScribe to integrate with any EHR that supports CDS Hooks (Epic, Cerner, etc.).

### Service Discovery

```
GET /cds/cds-discovery
```

Returns the service catalog including `medscribe-coding-review`.

### Hook: `encounter-discharge`

```python
@cds_service.hook(
    hook_type="encounter-discharge",
    id="medscribe-coding-review",
    title="MedScribe Coding Review",
    description="AI-assisted clinical documentation coding review",
)
def coding_review(request: CDSRequest) -> CDSResponse:
    ...
```

### Request Flow

```
EHR → POST /cds/cds-services/medscribe-coding-review
    → CDSRequest with prefetch containing DocumentReference
    → Extract clinical note text from prefetch
    → MedScribeAgent.run(note_text, patient_id) → result dict
    → CDSResponse(cards=result["cds_cards"])
```

### Prefetch Handling

The `_extract_note_from_prefetch()` function extracts clinical note text from CDS Hooks prefetch data:

1. Looks for `prefetch["document"]` as a FHIR `DocumentReference`
2. Calls `healthchain.fhir.read_content_attachment()` to get attachment data
3. Returns the `"data"` field from the first attachment
4. Returns empty string if anything is missing

### CDS Response Cards

The response contains CDS Hooks cards generated by the agent. Each card has:

- **summary:** Condition text and ICD-10 code (e.g., "Pneumonia -> J18.9")
- **indicator:** `info`, `warning`, or `critical` based on confidence and gaps
- **detail:** Documentation gaps and resolution reasoning
- **source:** "MedScribe Agent"

## Running With a FHIR Gateway

To enable live FHIR searches during gap resolution (Step 4), pass a configured `FHIRGateway`:

```python
from healthchain.gateway import FHIRGateway

fhir = FHIRGateway()
fhir.add_source("epic", "fhir://fhir.epic.com/r4?client_id=...")

app = create_app(fhir_gateway=fhir, fhir_source="epic")
```

When a gateway is configured:
- The `ToolRegistry` will execute real FHIR searches instead of using mock data
- The gateway is also registered with the HealthChainAPI for FHIR endpoints

Without a gateway (the default), tool dispatch uses mock data pools — see [Agent: Tool Dispatch](agent.md#tool-dispatch).
